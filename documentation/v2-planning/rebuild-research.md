# How supply chain security tools detect threats in open-source packages

**Modern supply chain security has split into two fundamentally different paradigms: reactive CVE scanning and proactive behavioral analysis.** Tools like Socket.dev and Phylum analyze what packages *actually do*—network calls, filesystem access, shell execution—to catch malicious code before any CVE exists. Traditional SCA tools like Snyk and Dependabot match dependency versions against known vulnerability databases, excelling at cataloging disclosed flaws but blind to intentional malware. The most robust defense layers both approaches, and a growing ecosystem of open-source tools makes building a custom supply chain scanner feasible for any engineering team.

## Deep package inspection versus CVE lookup

Socket.dev, founded by npm maintainer Feross Aboukhadijeh, pioneered "deep package inspection"—a deliberate analogy to deep packet inspection in networking. Rather than checking packages against vulnerability databases, Socket's custom static analysis engine examines the **actual behavior** of every package version across 10+ ecosystems (npm, PyPI, Go, Maven, Cargo, Ruby, NuGet, and others). The engine detects **70+ behavioral signals** organized around what code does at runtime: network access via `fetch()`, `http`, `dns`, or `net` modules; filesystem operations targeting sensitive paths like `~/.ssh` or `~/.npmrc`; shell execution through `child_process` or `subprocess`; environment variable reads that could steal credentials; `eval()` and dynamic code execution; and obfuscated or high-entropy strings suggesting encrypted payloads.

Socket's typosquatting detection combines **Levenshtein distance** with download count ratios—a package is flagged when its name is 1–2 characters from a popular package and the legitimate version has 1,000× more downloads. The tool `webb3`, for instance, was caught as a typosquat of `web3` based on a 300,000× download disparity. Install script detection is particularly critical: Socket flags `preinstall`/`postinstall` scripts in npm packages, especially when a previously script-free package suddenly introduces one—a hallmark of account takeover attacks. Maintainer analysis tracks ownership changes, chronological version anomalies (attackers publishing patches on old major versions with high usage), and ghost npm accounts.

Phylum, founded by U.S. Intelligence Community veterans and acquired by Veracode in January 2025, takes a complementary approach by combining static analysis with **dynamic sandbox execution**. Their automated engine processes every package published across seven ecosystems in real-time—**341 million files across 3.8 million packages in Q3 2024**—detecting malware in an average of **630 seconds** after publication. Where Socket relies primarily on static analysis, Phylum adds eBPF-based system call tracing inside sandboxed environments, correlating runtime behavior with static findings. Their ML models, trained on patterns from millions of packages, flag anomalies like brand-new packages that immediately access environment variables or spawn shell processes. Phylum identified nearly 500,000 malicious packages and 2,500 targeted campaigns before acquisition, including the first public report of North Korea's Jade Sleet nation-state campaigns in npm.

Sonatype takes yet another angle as stewards of Maven Central, using an **unsupervised ML anomaly detection model** that continuously learns from package features correlated with historical attacks. When the algorithm flags a suspicious component, the Sonatype Security Research Team conducts manual analysis, updating an "Integrity Rating" from suspicious to either malicious or normal. Their Repository Firewall concept is architecturally distinctive—rather than scanning at PR time, it intercepts and quarantines suspicious downloads at the repository proxy layer before packages ever reach developers.

## What CVE scanning catches and what it misses

Traditional SCA tools like Snyk, Dependabot, and Black Duck follow a well-established pattern: parse manifest/lockfiles, enumerate direct and transitive dependencies, match name+version tuples against vulnerability databases, and report known CVEs with severity scores. Snyk distinguishes itself with a proprietary database curated by a dedicated security research team, claiming to identify CVEs an average of **47 days before public databases** through monitoring GitHub commits, issues, social media chatter, and academic partnerships. Snyk is also a CVE Numbering Authority and added 24,000+ new vulnerabilities in 2024 alone. Its reachability analysis determines whether vulnerable functions are actually called in your codebase, reducing false positives by 60–80%.

The fundamental limitation is that CVE-based scanning is **reactive by design**. Supply chain attacks involve deliberately injected malicious code—not accidental vulnerabilities—and typically have no CVE at the time of attack. The `event-stream` incident of 2018 saw an attacker gain maintainer access to a package with 2 million weekly downloads and inject a targeted Bitcoin wallet stealer that went undetected for weeks. The `ua-parser-js` hijack in October 2021 installed cryptominers and password-stealing trojans on a package with 8 million weekly downloads, remaining live for four hours. In neither case would a CVE scanner have provided any warning during the active attack window.

The gap between these paradigms is concrete and measurable:

- **Typosquatting** (e.g., `crossenv` mimicking `cross-env`): no CVE exists for the fake package, but behavioral analysis detects suspicious name similarity plus malicious network calls
- **Dependency confusion**: no vulnerability in the package itself, but behavioral analysis catches install-time exfiltration to external servers
- **Install script attacks**: not classified as "vulnerabilities" by CVE databases, but immediately flagged by behavioral engines
- **Protestware** (like `es5-ext` geolocating users and showing protest messages): invisible to CVE scanners until manually reported and cataloged

The industry consensus is converging on a **layered defense model**: dependency updaters (Dependabot, Renovate) for hygiene; CVE scanning (Snyk, Grype, Trivy) for known vulnerabilities with reachability analysis to reduce noise; behavioral detection (Socket, Phylum) for unknown malicious packages; and repository firewalls (Sonatype Nexus, JFrog Xray) for organizational policy enforcement.

## Data sources that power the ecosystem

The vulnerability intelligence ecosystem rests on several interconnected databases. The **NVD** (National Vulnerability Database), operated by NIST, contains 250,000+ CVE records with CVSS severity scores and CPE product identifiers. Its REST API v2.0 at `services.nvd.nist.gov/rest/json/cves/2.0` supports search by CVE ID, date ranges, severity, and keywords. A free API key removes the default 6-second rate limit. The NVD's main limitation is enrichment lag—CVE records can take days to weeks after initial assignment before receiving complete analysis.

**OSV** (Open Source Vulnerabilities), operated by Google, addresses NVD's imprecision by mapping vulnerabilities directly to package versions and git commit hashes rather than CPE identifiers. It aggregates advisories from 24+ sources including GitHub Security Advisories, PyPA, and RustSec into a standardized JSON schema. The API requires no authentication: `POST https://api.osv.dev/v1/query` with a package name and version returns matching advisories, and batch queries handle up to thousands of packages. Full data dumps are available from the GCS bucket `gs://osv-vulnerabilities`.

The **GitHub Advisory Database** provides three tiers of advisories: GitHub-reviewed (curated, high quality), unreviewed (auto-imported from NVD), and malware advisories from the npm security team. These feed directly into `npm audit` and Dependabot. The REST API at `api.github.com/advisories` returns GHSA IDs, CVE cross-references, affected version ranges, and EPSS scores without authentication.

**deps.dev**, Google's Open Source Insights service, is perhaps the most underappreciated data source for builders. It covers 5+ million packages across npm, PyPI, Go, Maven, Cargo, RubyGems, and NuGet, providing fully resolved dependency graphs (using actual package manager resolution algorithms), license data, advisory mappings, OpenSSF Scorecard results, and SLSA provenance attestations. Its API requires no authentication and includes a particularly valuable **typosquatting detection endpoint** (`/v3alpha/systems/{system}/packages/{name}:similarlyNamedPackages`) and a **hash-based file lookup** for forensic identification. A BigQuery dataset enables large-scale analysis.

Package registries themselves provide critical metadata. The **npm registry API** (`registry.npmjs.org`) exposes maintainer lists, publish timestamps per version, install scripts, file counts, tarball sizes, and integrity hashes—all fields with direct security relevance. The `scripts` field (especially `preinstall`/`postinstall`) is the single most important npm malware vector. **PyPI's JSON API** (`pypi.org/pypi/{package}/json`) now includes a `vulnerabilities` field sourced directly from OSV. Download statistics from `api.npmjs.org/downloads/` and PyPI's BigQuery dataset provide the popularity signals needed for typosquatting detection.

## Open-source tools for building your own scanner

A rich ecosystem of open-source tools can be combined into a capable supply chain security scanner. These divide into distinct functional categories.

**For malicious package detection**, the OpenSSF's **package-analysis** project (github.com/ossf/package-analysis) provides a complete dynamic analysis pipeline. It monitors registries for new packages via the companion **Package Feeds** project, then "detonates" them inside **gVisor sandboxes**, capturing strace and network packet data. For Python packages, it goes further by importing modules and calling functions to trigger hidden malicious code. Results feed into the OpenSSF Malicious Packages Repository (15,000+ confirmed malicious packages in OSV format). The entire system can run locally with Docker for one-shot analysis.

DataDog's **GuardDog** (github.com/DataDog/guarddog) takes a static approach, implementing malicious package detection as **Semgrep rules** plus metadata heuristics across PyPI, npm, Go, Ruby, and even GitHub Actions and VS Code extensions. Its source code heuristics detect base64-encoded payloads flowing to `exec`/`eval`, download-and-execute patterns, environment variable exfiltration, and steganographic code extraction. Metadata heuristics check for typosquatting via Levenshtein distance, expired maintainer email domains, and repository integrity mismatches (comparing PyPI package contents against GitHub source). GuardDog is installable via `pip install guarddog` and outputs SARIF for GitHub Code Scanning integration.

**Packj** (github.com/ossillate-inc/packj) combines three analysis layers: static code analysis detecting sensitive API usage (based on Georgia Tech's MalOSS research), strace-based dynamic tracing, and metadata analysis including author email expiry and 2FA status checking. It supports PyPI, npm, and RubyGems.

**For static analysis engines**, **Semgrep** (github.com/semgrep/semgrep) is the most versatile foundation. Its open-source Community Edition supports 30+ languages with custom YAML rules for pattern matching and taint tracking. GuardDog's heuristics are built as Semgrep rules, demonstrating its suitability for supply chain patterns. GitHub's **CodeQL** provides deeper semantic analysis treating code as a queryable database, though its licensing restricts some commercial uses. **Bandit** (github.com/PyCQA/bandit) offers Python-specific security linting with 47+ plugins detecting `eval()`/`exec()` usage, insecure subprocess calls, pickle deserialization risks, and even AI-specific supply chain risks like unsafe `torch.load()` (plugin B614) and insecure Hugging Face downloads (B615).

**For vulnerability scanning**, **OSV-Scanner** (github.com/google/osv-scanner) scans lockfiles, container images, and SBOMs against the OSV database with guided remediation. Anchore's **Grype** (github.com/anchore/grype) scans containers and filesystems against NVD, GHSA, and distribution-specific databases. **pip-audit** (github.com/pypa/pip-audit, maintained by Trail of Bits) and **npm audit** handle ecosystem-specific vulnerability checking. Anchore's **Syft** generates SBOMs in CycloneDX and SPDX formats that feed into these scanners.

**For supply chain integrity**, the **OpenSSF Scorecard** (github.com/ossf/scorecard) automatically assesses project security practices across 18 checks including branch protection, code review, CI testing, dependency pinning, and SAST usage, scoring each 0–10. Pre-computed results for 1.25 million+ GitHub repos are available via API at `api.scorecard.dev`. **Sigstore** provides keyless artifact signing through ephemeral certificates (Fulcio) and a transparency log (Rekor), while the **SLSA framework** defines provenance levels—npm now supports SLSA Level 2 provenance natively with `npm publish --provenance`. **GUAC** (github.com/guacsec/guac) aggregates SBOMs, SLSA attestations, Scorecard results, and vulnerability data into a queryable graph database, providing a unified view of an entire software portfolio.

## Architecting a custom supply chain security system

The reference architecture follows a four-stage pipeline pattern, validated by both the OpenSSF package-analysis project and documented commercial architectures from SafeDep and others.

**The ingestion layer** monitors package registries for new and updated packages. For npm, the primary mechanism is the CouchDB changes feed at `replicate.npmjs.com/registry/_changes`, which provides a continuous stream of all package modifications. A follower process tracks sequence numbers and polls continuously. For PyPI, RSS feeds at `pypi.org/rss/updates.xml` (limited to 40 entries, polled every 30–60 seconds) are the recommended approach since XML-RPC is deprecated. Crates.io uses a git-based index; Go modules expose an index feed at `index.golang.org/index`; RubyGems offers activity feeds via its REST API. The OpenSSF **Package Feeds** project provides a ready-made implementation of multi-ecosystem polling with a unified event schema. All feeds should publish to a message queue (Kafka is recommended for its persistent log and consumer group support) using a normalized schema: `{ecosystem, package_name, version, timestamp, tarball_url}`.

**The analysis pipeline** should use a funnel approach to manage compute costs. **Stage 1 (metadata analysis, <1 second per package)** runs on every package: checking author/maintainer signals, version patterns, Levenshtein distance for typosquatting, dependency tree anomalies, and publication timing. This stage is implementable with PostgreSQL queries against stored metadata and a rule engine like OPA. **Stage 2 (static code analysis, 5–30 seconds)** runs on packages flagged in Stage 1 plus random sampling: AST parsing with language-specific parsers (acorn for JavaScript, Python's `ast` module, tree-sitter for multi-language), Semgrep rules for pattern matching, YARA rules for known malware signatures, entropy analysis for obfuscation detection, and optionally LLM-assisted classification of suspicious code blocks. **Stage 3 (dynamic analysis, 30–120 seconds)** runs only on high-risk packages: install the package inside a gVisor or Firecracker sandbox, monitor system calls via strace or eBPF, capture network traffic, and log process spawning. **Stage 4 (vulnerability correlation)** runs on all packages: batch-query a local OSV mirror plus the OpenSSF malicious-packages database.

This funnel dramatically reduces infrastructure requirements. With npm publishing roughly 2,000 new versions daily and all ecosystems combined producing 3,000–5,000 events daily, Stage 1 needs only a single worker (100+ packages/second capacity). After filtering, perhaps 500 packages daily need Stage 2 (3–4 workers), and 100 need Stage 3 (2–3 dedicated sandbox workers). A production deployment handling all major ecosystems is achievable on a Kubernetes cluster costing **$2,000–5,000/month**, with a minimum viable single-server deployment possible at ~$200/month covering npm and PyPI with Stages 1, 2, and 4 only.

**The storage layer** centers on PostgreSQL for structured data (packages, versions, analysis results, vulnerabilities, alerts), with JSONB columns for flexible findings storage. S3-compatible object storage (MinIO for self-hosted) stores package tarballs, raw strace logs, and generated SBOMs. Redis handles caching, job deduplication, and rate limiting. Elasticsearch enables full-text search across package metadata and fast Levenshtein-based typosquatting queries.

**The integration layer** is what makes the system actionable. A GitHub App that scans manifest changes on every PR and leaves inline comments is the highest-impact integration—this is Socket's primary workflow. A CLI tool wrapping package managers (`npm install`, `pip install`) can check packages before installation. A REST API should expose package risk scores, version-specific findings, bulk lockfile scanning, and SBOM generation. Webhooks to Slack, email, and PagerDuty route alerts by severity. For organizations requiring stronger guarantees, an HTTP proxy intercepting package manager requests can quarantine suspicious downloads at the network level—the Sonatype Repository Firewall model.

For sandboxing technology, **gVisor** is the recommended starting point—it's what OpenSSF uses, integrates natively with Kubernetes via RuntimeClass, and intercepts syscalls in userspace. For production hardening against sophisticated malware that can detect containerized environments, **Firecracker microVMs** provide full hardware-level isolation with ~125ms boot times. SafeDep's architecture documentation advocates for **eBPF over strace** for syscall monitoring because it hooks at the kernel level, making it harder for malware to detect and evade.

## Conclusion

Building a supply chain security scanner is no longer a moonshot—the open-source tooling has matured to the point where combining GuardDog's Semgrep-based heuristics, OpenSSF's sandboxed package analysis, OSV's vulnerability data, and deps.dev's metadata creates a system comparable in capability to early versions of commercial tools. The critical architectural insight is the **funnel pattern**: cheap metadata checks on everything, expensive dynamic analysis only on flagged packages. The critical conceptual insight is that **CVE scanning and behavioral analysis solve fundamentally different problems**—one catches known bugs, the other catches intentional attacks—and both are necessary. Any team serious about supply chain security should implement both layers, whether through commercial tools or this open-source stack. The biggest remaining challenge is not detection but triage: distinguishing legitimate telemetry from command-and-control traffic, acceptable minification from malicious obfuscation, and normal build-tool activity from data exfiltration remains an area where human judgment and ML-assisted classification are both still essential.
